MVP架构项目有何意义——实战

杨顺发
2017-01-12
MVP/architecture

来来来，2017年的第一发，也是第一次认真的写技术博文，想想还是有点小基动啊。我在想，要用以前那种闷骚逗比的文风来写，还是用一本正经的搞学术研究的样子来呢？嗯，我爱咋写咋写，不服打我哈哈哈哈哈。

咳咳，进入正题。

#MVP
MVP，即 Most、Valuable以及Player的缩写，指“最有价值的球员”，嗯，这是它的另一种含义。在软件编程这块，它的意思应该是“Model/View/Presenter”，具体的解释我就不对着维基百科的解释照抄了。这里着重说下Presenter，P 持有 M V 的对象，可以理解成 P 把控着它们，该取数据了，拿着 M 去取，取完得显示了，告诉 V 去显示。这个过程中 M V 是没有直接调用的，即我们说的”完全解耦“。基础就说这么多。

#深入
如果说MVP的好处仅仅只限于上述的代码解耦直观什么的，显得有点底气不足。后来在简书上看到文章《解读Android官方MVP项目单元测试》，才发现那个示例项目中的单元测试部分。在此之前，我一直都认为“没时间写测试代码“（有没有必要写单元测试不是这篇文章的主题，关于这点也是编程中一直有争论的一点），用真机测试手动点就好啦。看完那篇博文后，我对MVP有了新的认识：互相隔离的代码对于单元测试的支持非常好。既然做到了完全解耦，那么我就可以利用 Mockito和Espresso等测试支持库单独的测试Presenter，单独的测试View（炒鸡强大的Espresso测试其实已经覆盖了P和M）。

#测试Presenter
好的，根据《解读Android官方MVP项目单元测试》以及其他参考资料的学习，基本掌握了Mockito和Espresso的使用。于是在自己的项目中对Presenter进行测试。Presenter中不显示 UI ，也不获取数据，它只负责调度M和V，换言之，P只有业务逻辑。这就好办了。我不需要知道View中如何显示数据，也不care Model中如何获取数据，那怎么来玩儿？嗯，前边提了好几次的Mockito就能玩得666。创建P时通过构造函数注入了M和V，于是我们在测试类中可以通过Mockito去”Mock“M和V，及模拟出假的对象注入到P中，在执行过程中不会真正执行到 mock 出来的M和V中的实际代码。见栗子：

有些人认为写 UI（View）的测试代码没必要，不如直接用手操作来得快。我不晓得表达这些观点的人是否用过 Espresso ，也不晓得他们所在公司的项目是为何种。我只知道，在我学习并在公司项目使用它后，发现调试更加简单（直接运行Test既可Debug，并且可以模拟Intent直接启动目标Activity），测试的涵盖情况也更广（用户输入不同数据，得到接口不同的结果等等），不再需要手动点击各种各样的步骤进入指定页面，着实提高了我的 copy code 啊不，书写优质 code 的效率。

#测试Model
这块测试在我的项目中并不是那么突出。可能是使用 RxJava，固定的数据流请求接口得到的数据直接封装成 java bean 处理完并返回到P层中使用。所以这块并没有着重进行测试。而 Google 的官方栗子中也并没有 Model 层的测试代码。毕竟 Espresso 涵盖了。

#总结
写到这儿，这篇文章的所要表达的点就是 UnitTest 了。对于 UT 该不该写，一直都存在争论。在此之前，我也并不重视 UT，理由也是“没时间写”——分明就是懒嘛，毕竟学习这个东西也是需要时间成本的。学习完并应用到项目中后，才真实的感受到测试代码对效率的提升。

话说，这篇文章始于一月十二日，那天晚上写完，第二天就直接丢了。现在已是二月十四日，足足拖了一个月啊。虽然这期间一直在折腾内存优化这块。过了一个月，文章衔接度很差了，本来是用自己的项目作为栗子，测试View这块开始却直接使用了 Google 官方栗子，惭愧惭愧。
