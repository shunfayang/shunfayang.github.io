    前言

    几年前面试，我都喜欢去网上搜一下“Android 面试大全”“Android 面试宝典”，可是大全和宝典看得再多，面试过程却始终不尽如人意，最后的结果大多是“等通知”。近几年针对很多知识逐个研究击破，然后我也想造个轮子——整理一篇我自己的面试宝典。我先梳理了大致的框架，有基础部分、进阶部分和其他的一些广度知识，然后不断地添加其中的一些细节，比如生命周期，启动模式。在不断的填充以及学习之后，我逐渐在脑中形成了一个框架，仿佛这些知识彼此之间有一根线，连接彼此，形成一张巨大的知识网：生命周期，在不同的启动模式下，会有怎么的回调过程，在 ams（ActivityManagerService）中，又是如何实现的？关键的点在于，整篇知识总结的每一个字符，都是我在通过大脑思考后写下，抑或不懂的，学习之后总结出来的一些干练的句子。往后我都会不断取翻阅，翻阅的过程中又不断的补充，纠正。我恍然大悟，所谓的面试宝典，不过是作者对于自己的知识的总结，作者自己看到时，脑子里满是当时花好几个小时甚至好几天钻研的过程，只需要几句话几段话，便能将他带回记忆的片段重温学习的过程。而你，看着几行干瘪瘪的句子，感觉自己学会了，然后在跟别人探讨时，却发现自己在背诵，却说不出来为什么，却不知这些知识彼此之间的联系，泛泛而谈。所以啊，我希望这篇“面试宝典”会是你最后一篇面试宝典。关掉我的文章，打开你的 markdown，梳理你的知识，补充你的知识，让你的知识形成框架，然后在框架之中不断填入新掌握的知识，最后形成一张强大的网，这样，你才是真正掌握这些知识。

    分割线

    Android 基础

    • Activity 生命周期

      • onCreate 是必须实现的，需要做一些数据设置绑定的操作；
      • onStart 用户可见，进程进入前台，可以做一些支持交互的逻辑。比较快完成，此回调结束后会立刻进入下一阶段；
      • onResume 应用与用户交互阶段。如果启动一个透明 dialog 或者回到 Launcher 后重新进入 activity，会重新回调此方法；
      • onPause 此时失去焦点，成为后台进程。如启动一个透明 dialog，会回调，关闭 dialog 后，又会回到 onReusme；
      • onStop 回到主页或者别的非透明页面会回调，此时不再可见。可以做一些暂停动画，调整或释放不可见时的无用资源。这里也可以做一些持久性存储的工作，耗时可以长点，与 onStart 相反。
      • onDestroy 销毁 Activity 前调用。finish Activity 或者配置变化（横屏：销毁后立刻重建新对象执行新 onCreate）。这里应该释放所有资源（onStop 释放一部分）。
      • onRestart 从另一个非透明界面重新进入，会回到此方法，然后回调 onStart -> onResume。

    • Activity 启动模式

      • singleTask 是栈内只有一个目标 Activity，如果已经存在，直接复用，如果不是在最顶端，就把目标 Activity 上面的 activity 清空。
      • singleTop 就是顶部有就复用，没有就新建。

      • View 动画
      • 帧动画

    • View 事件分发机制

      • requestDisallowInterceptTouchEvent(true), 请求父控件不拦截事件，会直接使得不执行 onInterceptTouchEvent()。
      • 点击屏幕-ams-activity-ViewGroup-View—处理—>view.onTouchEvent()
      • 点击屏幕-ams-activity-ViewGroup-View—不处理—>ViewGroup.dispatchTouchEvent()—-不处理—->继续往上直至无人处理。
      • 扩展：Behavior

    • 自定义 View 绘制流程
      onMeasure：测量宽高，需要单独处理 wrap_content，不然与 match_parent 效果一致；
      onDraw：画圆画圈等绘制，需要深入复习 canvas 和 Paint 的功能；
      onLayout：集成 ViewGroup 需要重写这个方法，用于布置子 View 的位置，可以参看 LinearLayout 源码。

    • 自定义 View 优化
      如 Rect.clipRect，应该是这个，可以优化绘制区域；onDraw/onMeasure/onLayout 中不 new 对象，不打印 log；动画使用属性动画，不直接使用 Thread，可控。
    • 自定义 View 对 wrap_content 的讨论
      • 自定义 View 不处理 wrap_content 的情况下，不管父控件是 wrap_content 还是 match_parent，显示的效果都是撑开父控件到最大，换言之就是，都是 match_parent 的效果，都是占用父控件的最大宽度（或高度）。参看 TextView 的 onMeasure 实现，根据 padding、文字宽高、最小宽高等算出来一个值。甚至我可以理解成，wrap_content 就是需要具体控件去给出一个值。
    • Handler 机制-原理-源码
      • 核心：队列。主线程轮询队列，子线程往队列里添加数据。
    • 存储相关：数据库、文件

    • 事件分发更深层次原理：Input。本质上，Input 是一种消息，毕竟 Android 中的主线程是消息驱动的。屏幕有采样率，一般屏幕的采样率是 120Hz，当然现在 Android 主流已经是 120Hz 的刷新率，采样率更高了。当用户手指点击屏幕时（包含触摸和Home 等按钮），InputReader 会收到监听消息，读取到的 Input 消息会交给 InputDispatcher 进行消息分发；注册过的 App，其实就是当前屏幕上显示的 App，会收到 Input 消息，然后开始进行最最常见的 “事件分发机制”，其实就是把 Input 消息给到 ViewRootImpl，然后给到 Activiy 中 View 的根节点——DecorView。事件分发过程中，View 如果变化，又会去请求 Vsync 进行一帧的绘制，见文章2。

    • 应用启动流程

      1. 假设在 Launcher 中点击微信
      2. Launcher 所在进程会拿着微信相关的信息（如包名）通过 ams 的代理 binder 对象去调用 startActivity
      3. 代理 binder 访问 binder 驱动从 systemServer 中获取的 ams 在公共空间里的引用，最终调用到 ams 中 startActivity
      4. ams 让调用方进入 onPause
      5. ams 中进行一系列的逻辑后，判断是新启动的应用，那么就会通过 socket 通知 zygote 进程，fork 出一个新的应用的进程
      6. 新应用进程会调用 ActivityThread 的 main 方法
      7. 新应用进程在 main 执行后，将自己的 applicationThread 的 binder attach 到 ams，用于管理生命周期等
      8. ams 会通过新应用的 applicationThread 去创建 Activity 并调用 onCreate 等生命周期函数
      9. 新应用进入 resume 状态后，往 H 中添加了一个 idleHandler 通知 ams。

    • AIDL

      1. 创建 aidl 文件，扩展自己的方法
      2. 创建服务时，实现 aidl 生成的 .Stub（集成 Binder）并在onBind 时返回对象。
      3. 客户端获得 1 中的 aidl 文件，绑定服务时，使用 aidl 生成的 .Stub 的 asInterface（） 把 Binder 转成服务提供的 Binder 类型。
      4. 传递对象需要实现 Parcelable。
      5. 参考资料：Google 官方-Android 接口定义语言 (AIDL)

    • Binder
      研究了几篇文章下来，大致对 Binder 的认识如下：

      1. Server-Client 的机制
      2. 内存拷贝一次，发生在用户空间向内核空间拷贝的过程中
      3. ServiceManager 用于注册管理所有的 Server，访问 smgr 本身也是通过 Binder 通讯。
      4. Client 与 Server 访问的是内核空间中同一块地址
      5. 0 号引用预留给 ServiceManger。
      6. 一次拷贝，性能仅次于共享内存
      7. 参考资料：Android Binder设计与实现 - 设计篇、Android进程间通信（IPC）机制Binder简要介绍和学习计划
